# Model Equations & Logic — KM Trans Workshop Simulator

This document lists **every equation** in the simulator and the **reasoning** behind it,
tracing each one back to the KM Trans case exhibits. It is the technical companion to the
code in `js/` and is meant to be lifted into the project report.

Notation: capital letters are random variables; lower-case are parameters/data.
"Working minutes" = productive shop-floor minutes. A working **day** = `PMD = 420` minutes
(10:00–19:00 shift minus 2 hours of breaks → ~7 productive hours, per the case).

---

## 1. Why "working minutes" is the time unit

The case service times are already productive minutes:

```
Accident mean = 6300 min,  case says ≈ 15 days
6300 / 15 = 420 min/day  =  7 productive hours/day      ✔ consistent
```

**Logic:** because Exhibit 5 times and the 7-hour shift agree at 420 min/day, we run the
simulation clock purely in *working minutes* and convert to days with `days = minutes / 420`.
This sidesteps modelling nights/weekends explicitly while staying faithful to the case.
*(Code: `meta.productiveMinutesPerDay = 420`.)*

## 2. Arrival process (Exhibit 4)

### 2.1 Per-job-type arrival rate
Exhibit 4 gives, for each job type *i*, the number of arrivals over the 73-day window
(flatbed + car carriers). The **mean daily arrival rate** is:

```
λ_i  =  (flatbed_i + car_i) / 73           [jobs per day]
```

Total workshop arrival rate (with a user demand multiplier `dm`):

```
Λ  =  dm · Σ_i λ_i                          [jobs per day]   (≈ 30.4 at dm = 1)
```

### 2.2 Poisson arrivals → Exponential gaps
**Logic:** truck breakdowns/arrivals are many independent, individually-rare events → a
**Poisson process** is the standard, defensible choice. For a Poisson process of rate Λ, the
inter-arrival times are **Exponential**. Converting the daily rate to a per-working-minute rate:

```
λ_min  =  Λ / PMD                           [jobs per working minute]
```

Inter-arrival time sampled by **inverse-transform** (U ~ Uniform(0,1)):

```
T_gap  =  − ln(U) / λ_min
```

*(Code: `Distributions.exponential`, `sampleJobType`.)*

### 2.3 Choosing the job type and carrier class
Given an arrival, the job **type** is drawn in proportion to its rate, and the **carrier class**
(flatbed vs car) in proportion to its Exhibit-4 split:

```
P(type = i)      =  λ_i / Σ_j λ_j
P(carrier = flatbed | i) =  flatbed_i / (flatbed_i + car_i)
```

Keeping the carrier split lets us apply the case's **"car carriers > flatbed"** priority rule.

### 2.4 Over-dispersion check (use the data to justify the distribution)
Exhibit 4 also reports a **std. deviation of arrivals/day** per type. For a true Poisson
daily count, mean ≈ variance, i.e. `sd ≈ √λ_i`. Comparing the two is exactly the
"use the data to make the distribution" step the professor asked for:

```
Dispersion_i  =  sd_i / √λ_i
  ≈ 1   → Poisson is a good fit
  > 1   → over-dispersed (bursty); a Negative-Binomial daily count is more honest
```

We default to Poisson and **flag** types where `sd_i` is much larger than `√λ_i`
(e.g. Wiring, Pressure Leakage) in the report. *(This check is documented here and in the
report; the engine uses the Poisson default.)*

## 3. Service times (Exhibit 5)

### 3.1 Base mean and the skill-speed law
Exhibit 5 gives a mean time `m_i` for an **expert (skill level 10)** mechanic. The case states a
skill-*s* worker is slower in proportion to skill:

```
service-time multiplier  φ(s)  =  10 / s
   φ(10) = 1.0   (expert)
   φ(7)  ≈ 1.43  (medium)
   φ(4)  = 2.5   (low)         ← case: "low (4) rated → 2.5× expert time"
```

Skill-level mapping (Exhibit 2 categories → numeric skill): **High = 10, Medium = 7, Low = 4.**

When a job seizes several workers (possibly different skills), the **governing skill is the
minimum** among them — the job is not finished until the slowest sub-task is done:

```
s*  =  min( skill of all workers assigned to the job )
mean_effective  =  m_i · φ(s*)  =  m_i · 10 / s*
```

*(Code: `skillMultiplier`, `seizeWorkers` returns `minSkill`.)*

### 3.2 Lognormal service-time distribution
**Logic:** service times are positive and mildly right-skewed; a **Lognormal** captures this
(a Gamma would also work). The case says the coefficient of variation is small
(`CV ≤ 0.3`) except for **accident, denting, engine overhaul**, for which we use a heavier
`CV = 0.7`. Given a target mean `μ` and `CV`, the underlying normal parameters are:

```
σ²  =  ln(1 + CV²)
μ_log  =  ln(μ) − σ²/2
X  =  exp( μ_log + σ · Z ),     Z ~ N(0,1)
```

These follow from the Lognormal identities `E[X] = exp(μ_log + σ²/2)` and
`CV² = exp(σ²) − 1`, solved for `(μ_log, σ)` so the sampled mean equals the Exhibit-5 mean.

Standard Normal `Z` is generated by **Box–Muller**:

```
Z  =  √(−2 ln U₁) · cos(2π U₂),     U₁,U₂ ~ Uniform(0,1)
```

*(Code: `Distributions.lognormal`, `serviceTime`; `RNG.normal`.)*

## 4. Worker requirements & resource seizing (Exhibit 5 continued)

### 4.1 How many workers a job needs
Exhibit 5 (continued) gives, per job type and department *d*, a requirement `r_{i,d}`.
For ordinary jobs these are whole numbers (e.g. Clutch Overhaul = 2 Mechanical). The number
of workers seized is:

```
n_d  =  round(r_{i,d})         if r_{i,d} ≥ 1
n_d  =  1                      if 0 < r_{i,d} < 1   (fractional, accidents only)
n_d  =  0                      if r_{i,d} = 0
```

### 4.2 Fractional accident labour (time-share, not headcount)
The case explains the accident decimals as **time-share**: "a value of 0.1 for Balancer
showed that a worker was needed for only 10% of the job service time." We model this by
seizing one worker but **holding** it for only that fraction of the job duration `D`:

```
hold_d  =  r_{i,d} · D          if 0 < r_{i,d} < 1
hold_d  =  D                    otherwise
```

The **bay** is held for the full `D`; each department's workers are released at
`start + hold_d`. This reproduces the key case dynamic — an accident **blocks a bay** for the
whole duration while only fractionally drawing on each trade. *(Code: `startJob`, worker-release
events.)*

### 4.3 Start condition (seize rule)
A waiting job can **start** only when, simultaneously:

```
(a) a suitable bay is free          (see §5), AND
(b) for every department d:  available_workers_d  ≥  n_d
```

Otherwise it stays in the queue. On every event that frees a resource, the dispatcher retries.

## 5. Bays (case text, Exhibits 1 & 3)

Two pools: `generalBays` (the 8 numbered bays) and `extraBays` (the 4 planned long-job bays);
an optional idle **inspection bay** can be toggled on (+1 general).

```
separation ON :  job.bayType = "Extra"  → must use an extra bay (fallback to general only
                                           if extraBays = 0); all others use general bays.
separation OFF:  one pooled set of (generalBays + extraBays) serves every job.
```

"Extra"-type jobs (Accident, Denting, Cabin Setting, Tire — per Exhibit 4) are the candidates
for dedicated bays. **Logic:** separating long jobs is meant to stop a 15-day accident from
blocking a bay needed by dozens of 30-minute jobs; the toggle lets us *measure* whether it helps.
*(Code: `bayAvailable`, `seizeBay`, `releaseBay`.)*

## 6. Queue discipline & priority

The waiting queue is ordered before each dispatch attempt by:

```
1. Car-carrier priority (if ON):  car jobs rank ahead of flatbed jobs.
2. Then discipline:
     SJF  → ascending base mean service time  (case: "shortest job first")
     FIFO → ascending arrival time
```

*(Code: `dispatch` comparator.)*

---

## 7. Seasonal absenteeism (festival mode)

A festival absenteeism fraction `a` thins each department's roster, applied per skill bucket:

```
available_{d,s}  =  max( 0,  base_{d,s} − round(a · base_{d,s}) )
```

so e.g. `a = 0.4` removes ~40% of every department's workers for that run.
*(Code: `buildResources`.)*

## 8. Discrete-event mechanics (the DES loop)

State advances through a **Future-Event List (FEL)**, a min-heap ordered by event time.
Event types: `ARRIVAL`, `DEPARTURE` (bay + remaining workers released, stats recorded),
`WORKER_RELEASE` (fractional-accident workers freed early), `SNAPSHOT` (daily state capture).

```
while FEL not empty:
    e ← pop earliest event
    advance clock to e.time
    accumulate time-weighted queue area (§9.4)
    handle e  →  seize/release resources, schedule follow-up events
    run dispatcher to start any now-feasible jobs
```

**Logic:** event-scheduling DES only does work when something changes, so a 180-day horizon
with ~30 jobs/day runs in milliseconds — fast enough to be interactive in the browser.
*(Code: `runOnce`, `EventHeap`.)*

## 9. Output metrics

Let the **measured window** exclude a warm-up of `w` days:
`effMin = (simDays − warmupDays)·PMD`, `effDays = effMin/PMD`. Only jobs **arriving** after
warm-up are scored. For job *k*: arrival `a_k`, start `s_k`, completion `c_k`.

### 9.1 Waiting time and time-in-system
```
Wait_k        =  s_k − a_k                       (queueing delay)
Sojourn_k     =  c_k − a_k                       (total time in workshop)
avgWaitDays   =  mean_k(Wait_k) / PMD
P90 wait      =  90th percentile of {Wait_k}
```

### 9.2 Throughput
```
throughput/day  =  (number of completed jobs) / effDays
```
At a stable setting this equals Λ (what comes in goes out); when it falls **below** Λ the system
is overloaded and the backlog grows without bound.

### 9.3 Utilisation (clipped to the measured window)
Each occupancy interval `[start, start+hold]` is clipped to `[w·PMD, simDays·PMD]` so a job
spilling past the horizon only counts the minutes inside it — guaranteeing utilisation ≤ 100%.

```
busy_overlap = max(0, min(end, horizon) − max(start, warmupMin))

Bay utilisation     =  Σ bay busy_overlap / (totalBays · effMin)
Dept-d utilisation  =  Σ_d (workers · busy_overlap) / (capacity_d · effMin)
```

*(Code: `clip` in `startJob`; ratios in `summarize`.)*

### 9.4 Backlog (time-weighted average queue length)
We integrate queue length over time (area under the L(t) curve), Little's-law style:

```
avgQueueLen  =  ( Σ over events  Lₑ · (tₑ₊₁ − tₑ) )  /  effMin
```
where `Lₑ` is the number waiting between consecutive events. *(Code: `qArea`.)*

### 9.5 Delay cost
The case values a day of truck delay at **INR 2,500**. We charge avoidable **waiting** time:

```
DelayCost  =  Σ_k ( Wait_k / PMD ) · 2500     [INR]
```

This is the headline managerial number — what poor manpower/bay planning costs in rupees.

## 10. Replications & confidence intervals

A single run is one random sample, so we run `R` independent **replications** with seeds
`seed + r·7919` (a prime stride keeps the streams disjoint) and report the mean with a
**95% confidence interval** (normal approximation):

```
x̄  =  (1/R) Σ_r x_r
S   =  sample standard deviation of {x_r}
CI95 half-width  =  1.96 · S / √R
```

So a KPI shown as `x̄ ± CI95` means we are ~95% confident the true mean lies in that band.
A seeded RNG makes every result **reproducible** and makes scenario A vs B comparisons fair
(same random stream). *(Code: `run`, `aggregate`.)*

## 11. Validation against Exhibit 8

Exhibit 8 reports, for 340 real trucks, the share spending `< 1, > 1, > 2, > 5, > 20` days in
the workshop. We compute the **same buckets** from simulated `Sojourn_k/PMD` and overlay them
(as % of trucks) on the "Time-in-workshop vs case" chart. **Logic:** if the simulated
distribution reproduces the case proportions (≈ 91% under 1 day), the model is credible; large
gaps tell us which assumptions (CV, skill mapping, bay policy) to revisit. This is our reality
check, not a fitted parameter.

## 12. Assumptions register (state these in the report)

1. Poisson arrivals (over-dispersion flagged where Exhibit-4 `sd ≫ √λ`).
2. Lognormal service times; `CV = 0.3` routine, `0.7` for accident/denting/engine-overhaul.
3. Skill map High/Med/Low = 10/7/4; service time scales as `10/skill`; multi-worker jobs
   governed by the slowest (minimum-skill) worker.
4. Accident fractional requirements modelled as worker **time-share**, bay held full duration.
5. Spare parts always available (case: "shortage… generally not a major problem").
6. ~7 productive hours/day baked into the 420-min working day; nights/weekends not modelled
   explicitly (service times are already productive minutes).
7. Absenteeism applied statically per run; trivial (<30 min) jobs fixed outside the gate are
   excluded from the workshop model.
8. Delay cost charged on waiting time at INR 2,500/truck-day.
